# OpenFOAM 計算の現状課題と経緯

**作成日**: 2026-04-07
**対象**: SaunaFlow OpenFOAM (buoyantPimpleFoam) パイプライン

---

## 1. これまでの経緯

### 1.1 ソルバー起動までの9つのエラー

OpenFOAM buoyantPimpleFoam を初めて起動する過程で、以下の9件のエラーを順次修正した（詳細は `openfoam_troubleshooting.md`）:

1. `wallDist` 未定義（SST k-omega に必要）
2. `scalarCodedSource` の `name` 配置ミス
3. 圧縮性 div スキーム欠落
4. `div(phi,h)` 欠落（sensibleEnthalpy 形式）
5. FPE ゼロ除算（G_b の codedSource）
6. `rho` ソルバー未定義
7. `p_rgh` 初期値 0 → NaN（密閉キャビティの圧力参照不整合）
8. `externalWallHeatFluxTemperature` で初期発散
9. `deltaT` が大きすぎて発散

**教訓**: buoyantPimpleFoam の密閉キャビティケースは、圧力初期化・壁面BC・時間ステップの3点が同時に正しくないと起動すらしない。

### 1.2 計算は回るようになったが温度が合わない

#### 第1回: fixedValue 壁面 (T_wall = 20°C) — G_b 無効、300秒
- **結果**: 温度がどんどん冷える。T_upper: 68→45°C、T_lower: 54→32°C
- **原因**: 全壁面が外気温 20°C に固定されており、ヒーターとのバランスが取れない

#### 第2回: externalWallHeatFlux 壁面 (h=8 W/m²K, Ta=20°C) — G_b 無効、300秒
- **結果**: 成層は形成された (ΔT=13K) が、まだ温度が下がる傾向
- **原因**: 壁面熱抵抗を入れたが、ヒーターが `fixedValue T=593K` のためメッシュ解像度に依存する熱入力しか得られない

#### 第3回: 同上で 600秒に延長、nOuterCorrectors=4
- **結果**: T_upper: 62→34°C、T_lower: 45→27°C。冷却が続く
- **エネルギー収支分析で判明**: fixedValue ヒーター BC では M0 メッシュ (セル 0.125m) 上で **35W** しか熱入力できない。目標は **18,000W**。空気の分子熱伝導率 (0.026 W/mK) がボトルネック

#### 第4回: externalWallHeatFlux mode=flux (q=60,000 W/m²) に変更
- **結果**: 最初のタイムステップで h が発散、T = -347K で異常終了
- **原因**: 初期の乱流 (k=0.1) では乱流熱拡散率 αt が小さく、面熱流束がセルを直撃して温度が吹き飛ぶ

### 1.3 G_b (浮力生成項) の codedSource 問題

- `scalarCodedSource` はこの OpenFOAM 2312 で認識される
- しかし `wmake` (動的コンパイル) が Linux FS 上でも失敗する
- エラー: `exec(wmake, ...) failed`
- 原因未特定: コンパイラ環境 (gcc) の問題か、codedSource の C++ 構文の問題か

## 2. 現在の根本課題

### 課題A: ヒーター熱入力のメッシュ依存性

**問題の本質**: buoyantPimpleFoam では壁面からの熱伝達が「分子熱伝導 + 乱流熱拡散」で決まる。

| BC タイプ | 動作 | M0 での熱入力 | 問題 |
|-----------|------|-------------|------|
| `fixedValue T=593K` | セルと壁面の温度差 × kappa/dx | ~35W | 目標の0.2% |
| `mode=flux q=60000` | 熱流束を直接指定 | 18,000W (正確) | 初期ステップで発散 |

**flux BC が発散する理由**: 初期状態で k=0.1, omega=5 → nut=0.02 → alphat=0.024 → kappa_turb≈24 W/(mK)。これは十分な値だが、**最初の1ステップで h が更新された後、T→rho→p_rgh の連鎖更新で数値的に不安定になる**。

**試行中の対策**: k の初期値を 0.1→1.0 に引き上げ、乱流熱拡散を初期から強くして flux BC の衝撃を吸収する。

### 課題B: G_b codedSource の wmake 失敗

**問題**: `scalarCodedSource` の動的コンパイルが Linux FS 上でも失敗する。

**考えられる原因**:
1. OpenFOAM 2312 のビルド環境 (gcc, wmake) が不完全
2. `#include "fvCFD.H"` がヘッダーパスを見つけられない
3. WSL2 環境固有の問題

**影響**: G_b なしではサウナの温度成層が過度に弱化する（乱流混合を抑制する機構がない）。

**代替案**:
- `buoyantKEpsilon` 乱流モデルに変更（OpenFOAM 組込みの浮力モデル）
- `scalarSemiImplicitSource` で G_b を近似的に追加
- wmake 環境を修正して codedSource を有効化

### 課題C: メッシュ解像度 vs 計算コスト

| レベル | セル数 | セルサイズ | 計算時間 (600s) | 壁面境界層 |
|--------|--------|----------|-----------------|-----------|
| M0 | 9,600 | 0.125m | ~15分 | 未解像 |
| M1 | 76,800 | 0.063m | ~2時間 (推定) | 不十分 |
| M2 | 411,600 | 0.036m | ~12時間 (推定) | 最低限 |

サウナの自然対流境界層（厚さ数mm〜数cm）を適切に解像するには y+ < 1 が理想だが、M0/M1 では壁面関数に依存。壁面関数の精度が熱伝達率を左右する。

## 3. 簡易版との乖離の分析

| 指標 | 簡易版 (定常) | OpenFOAM (600s, G_b無し) | 乖離の原因 |
|------|-------------|------------------------|-----------|
| T_upper | 85°C | 34°C (冷却中) | ヒーター熱入力不足 |
| T_lower | 55°C | 27°C (冷却中) | 同上 |
| ΔT | 30K | 7.5K | G_b 無し → 乱流混合が過大 |
| z_interface | 1.03m | 不明瞭 | 成層が弱い |

**乖離の主因は ヒーター BC (課題A)**。簡易版は 18kW の対流熱入力を直接モデル化しているが、OpenFOAM の fixedValue BC では分子熱伝導がボトルネックとなり 1% 未満しか入力できない。

**G_b (課題B)** は ΔT に影響するが、そもそもヒーター入力が正しくないと比較の意味がない。

## 4. 次の一手（優先順）

### 最優先: ヒーター熱入力の正常化

1. **k=1.0 + flux BC + M1 メッシュ** で実行中
   - k を大きくして初期の乱流熱拡散を確保
   - M1 でセルサイズ半減 → 壁面関数がより正確
   - 結果を待って判断

2. **失敗した場合の代替**:
   - 体積熱源 (`scalarSemiImplicitSource` で h に直接エネルギー追加)
   - ヒーター周辺のメッシュのみ細密化 (grading)
   - 初期数ステップだけ低熱流束で漸増

### 次: G_b の有効化

1. `wmake` 環境の調査 (`which gcc`, `wmake -help`, OpenFOAM の `etc/config.sh/compiler` 確認)
2. 失敗なら `buoyantKEpsilon` に切り替え（壁面精度は落ちるが G_b は組込み）
3. または codedSource をプリコンパイルして .so を直接配置

### その後: 妥当性検証

1. ヒーター入力が正しくなった OpenFOAM 結果 vs 簡易版
2. 文献データ（サウナ室温度成層の実測値）との比較
3. メッシュ依存性確認 (M1 vs M2)

## 5. 技術的補足

### なぜ簡易版は正しく動くのに OpenFOAM は苦労するか

簡易版 (2-Zone) は:
- ヒーター出力を **直接** エネルギー保存式に投入（メッシュ不要）
- 壁面熱損失は Newton 冷却則で **解析的** に計算
- 質量保存は Zukoski 相関式で **パラメトリック** に解決

つまり、メッシュ解像度・壁面境界層・乱流モデルの影響を**全く受けない**。

OpenFOAM (3D CFD) は:
- 壁面からの熱伝達を**セルレベルの温度勾配**で計算
- 乱流モデルが壁面熱伝達率を支配
- メッシュが粗いと「熱の通り道」が狭くなる

この根本的な違いが、同じ物理パラメータでも大きく異なる結果を生む原因。

### externalWallHeatFluxTemperature mode=flux の正しい使い方

この BC は **kappa (熱伝導率) × 面積法線方向の温度勾配 = q** を壁面で強制する。つまり壁面セルの温度を q/kappa だけ壁面から離れた位置にずらす。

kappa = kappa_molecular + kappa_turbulent であり、乱流が発達していないと kappa_molecular (0.026 W/mK) だけになる。このとき q=60000 だと dT = 60000 × dx / 0.026 ≈ 290,000K。これが最初のステップで爆発する理由。

**解決策**: k の初期値を十分大きくして kappa_turbulent を確保するか、初期数ステップだけ q を小さくする。
