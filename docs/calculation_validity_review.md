# SaunaFlow 計算妥当性レビュー（客観評価）

**作成日**: 2026-06-27
**対象**: `src/harness/simple_solver.py`(2ゾーンプリュームROM), `kpi.py`, `validation.py`, および OpenFOAM 計算キャンペーン (Case A〜L)
**目的**: 現状の計算の妥当性を第三者視点で評価し、今後の方針・改善点を整理する

> 本レビューは批判のためではなく、「どこまでが信頼できる結果で、どこからがチューニング/仮定なのか」を明示し、PoC を次段階に進めるための材料にすることを目的とする。コードを実際に実行して得た数値根拠を併記する。

---

## 0. 総評

SaunaFlow の計算層は **2つの独立した計算系**から成る:

1. **2ゾーンプリュームROM** (`simple_solver.py`) — MTT/Zukoski をベースにした縮約モデル
2. **OpenFOAM CFD** — buoyantPimpleFoam による3D計算

両者とも「サウナの温度成層という支配的な現象の**定性的トレンド**」は再現できている。物理の参照文献(Morton-Taylor-Turner, Zukoski, Heskestad)は実在し妥当で、コード品質・テスト網羅(284関数)・自己ドキュメントも良好。**PoC として方向性は正しい。**

一方で、現時点で「定量的に妥当」と主張できる結果は存在しない。最大の理由は次の3点:

| 重大度 | 論点 |
|--------|------|
| 🔴 致命 | **検証の基準データが推定値**であり、モデルはその推定値に合うようキャリブレーションされている（循環論法） |
| 🔴 高 | **定常ソルバーがエネルギー収支を閉じておらず未収束**。報告温度は反復打ち切り時のスナップショット |
| 🟠 高 | **ロウリュに顕熱ピーク機構がなく**、KPI K-02 が捉えるべき現象を物理的に持たない |

以下、根拠とともに詳述する。

---

## 1. 妥当な点（評価できる箇所）

- **プリューム入力 (`_plume_entrainment`)**: Zukoski 相関 `m = 0.071·Q_c^(1/3)·z^(5/3) + 0.0018·Q_c` と Heskestad 仮想原点は火災工学の標準。実行結果も物理的に妥当（z↑で m_dot↑・T_plume↓）:
  ```
  z=0.3m: m=0.110 kg/s, T=191°C   z=2.0m: m=0.725 kg/s, T=94°C
  ```
- **成層トレンドの再現**: ROM/CFD とも upper>lower>floor の順序と 20〜40K オーダーの上下差を再現。
- **換気スタックモデル (`_ventilation_flow`)**: 浮力スタック圧 + オリフィス流れの組み立ては教科書的に妥当。上流密度の使い分けも正しい。
- **皮膚熱収支による体感温度 (`_perceived_temperature`)**: Steadman 式（20〜50°C限定）を捨て、対流+蒸発(Lewis関係)+放射の熱収支で 60〜120°C を扱う方針は適切。
- **パラメトリック手法**: 1軸ずつ変動させて因果を切り分けた方法論（`computational_insights_summary.md`）は健全。
- **エンジニアリング品質**: YAML宣言→build→run→parse の自動化、284テスト、絶対インポート、UTF-8徹底。

---

## 2. 妥当性の懸念（根拠つき）

### 🔴 C1. 検証データが推定値 — キャリブレーション循環（最重要）

`experiments/processed/dry_sauna_sample.csv` は実測ではなく推定サンプル。`computational_insights_summary.md` の「実測目標 upper 80-100°C」も推定。一方 OpenFOAM 側は **13kW volume source + fixedT 573K** をこの推定目標に合うよう選んでいる（知見4「合計~14.6kWが実測に合う」）。

つまり現状は **「推定値に合わせたモデルが推定値に合う」** ことを確認しているに過ぎず、`validation.py` の `compare_probes` が `overall_pass=True` を返しても**それは妥当性の証拠にならない**。本プロジェクト最大のリスク。

**これは既知**（docs 5.5「実測データの不在」）だが、レビュー上は最優先課題として明記する。

### 🔴 C2. 定常ソルバーがエネルギー収支を閉じず未収束

`solve_two_zone` を `dry_sauna_steady.yaml`(18kW) で実行した実測:

```
converged = False   iters = 10000(上限)   final residual = 1.17e-4 (> tol 1e-4)
T_upper = 88.9°C  T_lower = 58.5°C  wall = 55.5°C
エネルギー収支: 外部への壁伝導 12,081W / 入力 18,000W  → 閉合率 0.67
```

- **収束していない**: 残差がトレランスに到達せず10000反復で打ち切り。報告温度は擬似時間ステップ(`dt=0.5`陽解法)の途中状態。
- **エネルギーが約6kW不整合**: 定常なら「入力＝外部放熱」が成立すべきだが、外部放熱は入力の67%しかない。空気層が受け取る顕熱(プリューム12.6kW)に対し壁面対流損失は6.7kWで、差分が行き場を持たず系が定常に達していない。
- **影響**: `simple_solver` の報告値は「収束解」ではなく「打ち切り時のスナップショット」。パラメータを変えた時の比較が、物理的平衡の比較なのか過渡状態の比較なのか区別できない。

> 根拠スクリプト: `scratchpad/diag.py`（本ブランチの作業中に実行）。`l.793` の `np.clip(t_upper, t_wall_inner, t_wall_inner+200)` 等のクランプが、未収束でも発散せず「それらしい値」を返してしまう点が問題を見えにくくしている。

### 🟠 C3. ロウリュに顕熱ピーク機構がない → K-02 が現象を捉えない

`loyly_test.yaml` の過渡実行:
```
T_upper(乾球): start 21.0°C → 単調増加 → end 55.4°C  (ロウリュ由来のピークなし)
humidity peak = 41.6 g/kg,  perceived peak = 66.6°C
```

コード(`l.726-743`, `l.1069`)は「潜熱は石から引かれ、蒸気は湿度を上げるが正味の熱は加えない」という設計。結果、**乾球温度にはロウリュのスパイクが一切現れない**（上の実行で T_upper は単調増加）。

しかし KPI **K-02「ロウリュ後ピーク温度上昇」(`compute_k02`)** は乾球時系列の最大値を見る。現実のロウリュは「熱い湿った空気塊が一気に到達する短時間の顕熱+潜熱バースト」であり、体感の核心。現状モデルでは:
- K-02 は「ベースライン到達過程の最大値」を拾うだけで、ロウリュ現象を測れていない
- ロウリュの効果は湿度(K-03)と体感温度にしか現れない

**設計判断としては理解できるが、KPI 定義（K-02〜K-04）が想定する物理がモデルに無い**という不整合がある。

### 🟠 C4. View factor がヒューリスティックで物理的に逆の配分

`_compute_view_factors` は atan ベースの立体角近似に多数の `np.clip`/再正規化を重ねたもの。実行結果（床近傍 y=0.1 のヒーター）:
```
floor=0.226, lower_walls=0.104, upper_walls=0.545, ceiling=0.120, body=0.005
```
**床のすぐ上のヒーターが上部壁へ 54%** を放射する配分になっており直感に反する（近接する床・下部壁が支配的なはず）。総和=1.0 の閉包は満たすが、相反則(reciprocity)も放射熱平衡(radiosity)も満たさない**チューニング済みヒューリスティック**であり、「view factor」という名称は実態より強い。知見2が示す通り感度自体は低い(<1°C)ので実害は小さいが、名称と中身の乖離は是正すべき。

### 🟠 C5. Aufguss が定常ソルバーで「常時ON」= 非物理

`solve_two_zone` の `l.808-817`「In steady-state solver, aufguss is always active (no physical time)」。Aufguss は本質的に数十秒の過渡現象であり、定常解に常時適用するのは物理的に無意味。実測:
```
beta_aug=0.5 で上下差 30.3K → 8.7K（方向は正しいが大きさは入力 beta_aug の線形写像）
```
方向（混合で成層が崩れる）は正しいが、定常での扱いは過渡モデル(`solve_transient`の窓制御)へ寄せるべき。

### 🟠 C6. 一部 KPI がトートロジー（出力＝入力の再スケール）

- **K-05 顔面風速** (`compute_k05`): `v = beta_aug/(rho·0.05)`。`beta_aug` はユーザー入力の混合係数で、`a_face=0.05` も固定。**何も予測しておらず入力の線形変換**。「風速」を名乗るならジェット運動量モデルから導くべき。
- **K-01 pass/fail** (`compute_k01`): `diff>0` で pass。成層があれば常に通る＝**合否基準として無意味**。
- これらは「KPIが緑だから妥当」という誤った安心感を生む。

### 🟠 C7. クランプ多用が数値不安定を隠蔽

`l.792-794, 806, 816-817` 等で温度・界面高さを毎ステップ `np.clip`。発散を防ぐ実用策だが、**収束時にクランプが作動しているか否かを検査していない**。クランプ境界に張り付いた解は物理平衡ではない。「収束＝クランプ非作動」を assert すべき。

### 🟠 C8. OpenFOAM 側：メッシュ非依存性が未検証・単点キャリブレーション

`computational_insights_summary.md` 自身が認める通り:
- 定量結果は **L-1 の1ケースのみ**（M0=9,600セル, PIMPLE 3600s）。
- **メッシュ非依存性評価なし**（M0のみ、M1は計算コストで未実施）。
- SIMPLE は50k反復でも定常未到達（知見1）。
- volume source も「有効対流伝熱」としてのキャリブレーション値。

→ 離散化誤差が定量化されておらず、CFD 結果も単点・チューニング依存。

### 🟢 C9. fixed-wall モードで放射が空気を直接加熱

`l.720, 772` の `wall_cfg != "lumped"` 経路で `q_rad_to_walls*f_rad` を空気層に直接加算。空気は熱IRにほぼ透明なので物理的に誤り。既定の `lumped` では壁経由に正しく回しているので実害は限定的だが、`fixed` 経路は誤解を招く。

### 🟢 C10. 不確かさの定量化が皆無

`f_conv=0.7`, `h_wall_base=8.0`, 壁 λ など主要パラメータは固定の単一値。KPI も点推定のみ。これらの不確かさが結果にどれだけ効くかが示されておらず、「±何度の信頼区間か」を答えられない。

## 3. 今後の方針（優先度つきロードマップ）

### フェーズ A — 妥当性の土台を作る（最優先・コード変更小）

> **実装状況（2026-06-27, 本ブランチ）**: 本フェーズの自己診断を実装済み。`simple_solver.py` に `_energy_balance()` / `_clamp_active()` を追加し、`SimpleSolverResult` に診断フィールドを持たせ、`batch.py` のレポートに常時出力する。新規テスト9件を追加（既存挙動は不変）。

1. **エネルギー収支の診断を常時出力 (C2)** — ✅ 実装
   最終状態で「系に入る全パワー(=heater power) と境界から出る全パワー(壁伝導 or 壁対流 + 換気エンタルピー)」を計算し、`energy_in_w / energy_out_w / energy_residual_w / energy_closure` を `SimpleSolverResult` に格納。batchレポートに「Convergence & Energy Balance」表として表示。
2. **収束判定にクランプ検査を追加 (C7)** — ✅ 実装
   `_clamp_active()` で状態量がクランプ境界に張り付いているかを検査。既存の `converged`（数値残差の意味）は後方互換のため変更せず、新たに **`physically_converged = converged AND エネルギー閉合(±10%) AND クランプ非作動`** を追加。「数値的に収束＝物理的に妥当」という誤解を断つ。
3. **未収束の顕在化** — ✅ 部分実装（テストで固定）
   この診断で判明した重要事実: **テストが `converged=True` と判定する 9kW ケースですら energy_closure ≈ 67%**（18kW dry_sauna_steady も 67%、過渡2000sでも 59%）。残差が小さくてもエネルギーは閉じておらず、報告温度は壁面蓄熱が終わっていない過渡スナップショット。`test_numerical_convergence_can_hide_energy_imbalance` でこの事実を固定。
   - 残課題: `dt=0.5` 陽解法 + 巨大壁熱容量(≈32万J/K)が律速。真の定常到達には界面-上層-壁の連成を半陰的にするか `dt` 適応化が必要（フェーズC以降）。

### フェーズ B — 実測で循環を断ち切る（プロジェクト最重要）

4. **実測データ取得 (C1)**: DHT22 校正・設置（`measurement_implementation_plan.md` 既存）。**複数運転点**（異なる kW・換気設定）で取得し、1点フィットを避ける。
5. **キャリブレーション点と検証点の分離**: あるケースで合わせ、**別ケースで予測精度を検証**する hold-out 方式を `validation.py` に導入。`configs/acceptance/`（現在空）に運転点ごとの許容差を定義。

### フェーズ C — 物理の忠実度向上

> **実装状況（2026-06-27, 本ブランチ）**: C3・C5 を実装。C4/C9 は次項のとおり前提作業が必要なため未着手（理由を明記）。

6. **ロウリュの顕熱/潜熱を分離 (C3)** — ✅ 実装
   蒸気が石から運ぶ**顕熱**を上層エネルギー収支に追加（`solve_transient`: `q = m_steam · cp_vapor · (T_steam − T_upper)`）。潜熱は従来どおり石から引かれ空気には直接入らず湿度のみ上げる、という分離を維持。これにより従来「乾球温度に一切現れなかった」ロウリュが、注水直後に乾球のバンプを生む（実測: dry 比 +約2K、`steam_temp_c` を上げると増大）。蒸気温度は `loyly.steam_temp_c`（既定100℃、schema追加）で可変。新規テスト `TestLoylySensibleHeat` 3件で「顕熱が入る／高温ほど高い／潜熱の二重計上がない」を固定。
   - 注: 体・壁での凝縮潜熱放出（体感のさらなる上振れ）は未モデル化。体感温度は既存の `_perceived_temperature` 経路が担うため、K-02 は乾球、K-06 は体感、と役割を分離して解釈する。
7. **放射モデルの正名化 (C4/C9)** — ✅ 実装
   - **前提作業（フィクスチャ移行）**: 壁モデルの**既定を `fixed`→`lumped`** に変更（schema既定も更新）。`walls.model` 省略の全ケース（テストフィクスチャ含む）が物理パスで走るようになった。移行後も収束・成層・換気冷却・Aufguss の全テスト前提が維持されることを実測確認済み。
   - **C9（空気の直接放射加熱を廃止）**: 空気は熱IRにほぼ透明なので、ヒーター放射が**空気を直接加熱する経路を完全に削除**（`q_rad_to_upper/lower = 0`）。放射は壁へ届き、`lumped` では壁が昇温して空気へ対流戻し、`fixed` では定温シンクに吸収されて室を温めない（＝対流のみの極限ケースと明記）。不要化した `f_rad_upper/lower` を除去。新規テスト `TestRadiationModel` で「既定=lumped」「fixed は放射加熱なしで lumped より低温」を固定。
   - **C4（view factor の正名化）**: `_compute_view_factors` を **`_radiant_distribution` に改名**（旧名はエイリアス維持）。docstring で「相反則を満たす view factor ではない／radiosity の代替ではない」「閉包のみ enforce、相反則は非保証」と明示。さらに C9 で放射が壁分配を介さなくなったため、この分配の壁/床/天井成分は**実物理では未使用**（体への放射 `f_body` のみ消費）であることも明記。
   - 残（任意）: 厳密にやるなら小規模 N 面の radiosity（相反則＋熱平衡）へ置換。現状の PoC では不要。
8. **Aufguss を過渡専用に (C5)** — ✅ 実装
   `solve_two_zone`（定常）で `beta_aug>0` のとき**警告を出す**（「Aufguss は過渡現象。定常では時間平均サロゲートとして連続適用される。物理的結果は `solve_transient` の Aufguss 窓を使え」）。既存の定常 Aufguss 挙動は後方互換のため残置（テスト不変）。`solve_transient` は従来どおり時間窓制御。新規テスト `TestSteadyAufgussWarning` 2件。

### フェーズ D — 信頼性の定量化

> **実装状況（2026-06-27, 本ブランチ）**: C10・C6 を実装、C8 は解析ツールを実装（OpenFOAM 実行は外部）。

9. **メッシュ非依存性 (C8)** — ✅ 解析ツール実装（実行は外部）
   `grid_convergence.py` に Richardson 外挿 + Roache/Celik GCI を実装（`three_grid_gci`）。3 グリッド(M0/M1/M2)のプローブ値から観測精度次数 *p*・h→0 外挿値・GCI 誤差バンド・漸近領域判定を出す。**OpenFOAM の M1/M2 実行自体は WSL+OpenFOAM 環境で行う**もので本ツールはその出力（各グリッド1スカラー）を消費する。テスト `test_grid_convergence.py`(5件、p≈2 の合成データで検証)。
10. **感度・不確かさ (C10)** — ✅ 実装
   `sensitivity.py` に局所感度（中心差分、絶対 dKPI/dp と無次元エラスティシティ）と線形不確かさ伝播（σ_KPI=√Σ(∂KPI/∂p·σ_p)²）を実装。CLI `saunaflow sensitivity` で YAML 露出パラメータを振り、KPI を**点推定でなく ±2σ バンド**で報告。テスト `test_sensitivity.py`。
   - **ハードコード定数のパラメータ化（追補）**: レビューで名指しの `f_conv`・`h_wall` を YAML 化（`heater.convective_fraction` 既定0.7、`walls.h_natural` 既定8.0、schema追加）。既定値は従来のハードコード値と同一で**結果不変**。これで両者も感度・不確かさ解析の対象になり（CLI 既定候補にも追加）、`λ`・壁厚と合わせ主要パラメータが一通り振れる。`TestParameterization` で方向性（対流率↑→昇温、h_natural↑→冷却）と既定一致を固定。
   - 注: 非収束ケース(C2)では感度の符号が物理直感と逆に出ることがあり、**C2 の収束診断と併読する前提**。
11. **KPI の健全化 (C6)** — ✅ 実装（K-05 は注記、ジェット化は将来）
   `KPIResult.note` を追加し、proxy/トートロジー/前提を明示。**K-01** を `diff>0`(自明)から**妥当性バンド(2–80 K)**に変更（well-mixed/非収束を fail+注記で検出）。
   - **K-05 のジェットモデル化（追補）** — ✅ 実装: K-05 を mixing係数の再スケールから、**乱流自由噴流の中心速度減衰モデル**（`harness/jet.py: free_jet_face_velocity`）へ置換。Aufguss の物理パラメータ `jet_velocity`(U₀)・`jet_diameter`(d₀)・`face_distance`(x)（schema追加）から、ポテンシャルコア(`x≤6.2 d₀`)では U₀、以遠は `U₀·6.2 d₀/x` で顔面風速を予測。ソルバーが `aufguss_face_velocity` を算出し結果に格納、K-05 はそれを採用して**知覚閾値(0.2 m/s)で pass/fail**。ジェット値が無い場合のみ従来 proxy にフォールバック（注記つき）。テスト `test_jet.py`＋K-05/ソルバー結線テスト。
